# 작업 규칙

## 1. 완료 상태 구분
모든 작업 보고는 아래 4개 상태 중 하나로만 보고한다.

- 작성만 됨
- 서버 반영됨
- 실행됨
- 검증완료

이 외의 표현으로 완료처럼 보고하지 않는다.

## 2. 완료 보고 금지 조건
아래 중 하나라도 없으면 완료라고 보고하지 않는다.

- 서버 반영 확인
- 실제 실행 명령어
- run_id 또는 실행 식별값
- 호출 로그 수치
- DB 적재 결과
- 샘플 검증 결과

## 3. 미실행 규칙
실행하지 않았으면 첫 줄에 반드시 `미실행`이라고 명시한다.

코드만 작성한 상태를 완료처럼 설명하지 않는다.
설계상/예상상/구조상이라는 표현을 실제 실행 결과처럼 사용하지 않는다.

## 4. 메모리 보고 금지
다음 표현을 보고에 사용하지 않는다.

- 메모리에 저장했다
- 기억했다
- 저장해두었다

사용자 작업에서 필요한 것은 메모리가 아니라 로직 반영, 서버 실행, DB 적재, 검증 결과다.

## 5. 숫자 없는 정상 보고 금지
아래 없이 정상/완료/반영이라고 보고하지 않는다.

- 호출 수
- 성공 수 / 실패 수
- 적재 건수
- 샘플 검증 결과

## 6. 실행 우선 규칙
다음 단계가 명확하면 사용자에게 다시 묻지 말고 바로 실행한다.
단, 실제로 막히는 항목이 있을 때만 보고한다.

막히는 경우 반드시 아래를 함께 보고한다.
- 무엇이 막혔는지
- 왜 막혔는지
- 어디서 막혔는지
- 현재 가능한 대안

## 7. 개발/배치 작업 보고 형식
모든 개발/수집/배치 작업은 아래 순서로 보고한다.

### A. 반영 상태
- 수정 파일 목록
- 서버 반영 여부
- 마이그레이션 여부

### B. 실행 상태
- 실행 명령어
- 실행 시작/종료 시각
- run_id
- 성공/실패 여부

### C. 검증 상태
- 호출 로그 수치
- DB 적재 결과
- 샘플 검증 결과

## 8. 수집 작업 필수 검증
수집 로직 변경 후에는 반드시 아래를 제출한다.

- 실제 호출 수
- 지역별 호출 수
- 업종별 호출 수
- 실패 조합 수
- 적재 건수
- 샘플 공고 검증 5건 이상

## 9. 추론과 사실 구분
다음을 명확히 구분해서 표시한다.

- 추론
- 코드상 확인
- 서버 실행 확인
- DB 검증 확인

추론을 사실처럼 보고하지 않는다.

## 10. 사용자 작업 성격
이 프로젝트는 실운영 데이터 수집/분석 작업이다.
따라서 코드 작성보다 실행 검증이 더 중요하다.
검증되지 않은 결과를 완료처럼 보고하는 것을 금지한다.

## 11. 질문 최소화
- 다음 단계가 명확하면 묻지 말고 바로 진행
- "다음으로 무엇을 할까요?" 같은 문장 금지
- 멈춰야 하는 경우는 실제 차단 요소가 있을 때만

## 12. 보고 문장 제한
다음 표현은 증거 없이 사용 금지:
- 완료되었습니다
- 정상입니다
- 반영됐습니다
- 검증 완료입니다

사용 가능 조건: 실행 명령어 + run_id + 로그 집계 + 적재 건수 + 샘플 검증 결과 중 해당 항목 존재 시.

## 13. 미실행 상태 문구 강제
실행 전 보고 첫 줄은 반드시 아래 중 하나로 시작:
- `미실행`
- `서버 반영 전`
- `코드만 작성됨`
- `실행 준비 완료`

## 14. 수집/배치 작업 5단계 강제
데이터 수집 작업은 반드시 아래 5단계를 모두 거쳐야 한다.
1. 코드 반영
2. 서버 반영
3. 실제 실행
4. 로그/DB 검증
5. 샘플 검증

이 중 하나라도 빠지면 완료라고 보고하지 않는다.

## 15. 샘플 검증 의무 (소방 입찰/개찰)
소방 입찰/개찰 분석 작업에서는 최소 샘플 5건 이상 대조 제출.
검증 항목: 공고번호, 지역명, 업종명, 기초금액, A값 여부, 예정가격, 개찰 연결 여부, 계약 연결 여부.

## 16. 막힘 보고 형식
막힐 경우 아래 형식으로만 보고:
- 막힌 작업:
- 막힌 이유:
- 막힌 위치:
- 현재 영향:
- 우회 가능 여부:
- 즉시 가능한 다음 작업:

## 17. 메모리 행동 비중 축소
개발/운영 작업 중 메모리 저장은 우선순위가 아니다.
메모리 저장 여부를 보고하지 말고, 실제 코드/실행/검증 결과만 보고한다.

## 18. admin API audit/log 작성 규칙 (FK 선조회 원칙)

신규 admin API route에서 audit/log/history 기록 시 반드시 아래 순서를 지킨다.

### 필수 순서
1. `findUnique` / `findFirst` 로 FK 대상 row 먼저 조회
2. 권한/분기 처리 (PERMISSION_DENIED 등)
3. row가 존재할 때만 audit create 실행

### 금지 패턴
```typescript
// ❌ 금지 — params.id를 DB 미검증 상태로 audit에 직접 사용
if (session.role === 'VIEWER') {
  await logPresenceAudit({ presenceCheckId: params.id, ... })
}
const pc = await prisma.presenceCheck.findUnique(...)
```

### 필수 패턴
```typescript
// ✅ 올바른 순서 — DB 조회 먼저, 존재 확인 후 audit
const pc = await prisma.presenceCheck.findUnique({ where: { id: params.id } })
if (session.role === 'VIEWER') {
  if (pc) await logPresenceAudit({ presenceCheckId: pc.id, ... })
  return forbidden(...)
}
```

### 적용 대상
- `logPresenceAudit` 호출 위치 전체
- `writeAuditLog` / `writeAdminAuditLog`의 `targetId`가 FK인 경우
- confirm / reject / reissue / approve / deny / suspend / delete 계열 route

### 헬퍼 사용
`logPresenceAuditSafe(presenceCheckId, params)` 를 사용하면
내부에서 존재 확인 후 기록하므로 순서 실수 방지 가능.
(단, 이미 선조회한 pc 객체가 있으면 `logPresenceAudit({ presenceCheckId: pc.id })` 직접 사용이 더 효율적)

## 19. ops-check 기준선 운영 규칙
- 장애 발생 / 배포 직후 / 정기 점검 시 `bash scripts/ops-check.sh` 를 가장 먼저 실행한다
- 결과는 PASS/WARN/FAIL 요약만 보고한다
- FAIL 시 자동 첨부된 로그를 확인하고, 승인 전 재시작/삭제/롤백 금지 원칙을 유지한다
- ops-check 스크립트 경로: `scripts/ops-check.sh` (서버: `~/app/attendance/scripts/ops-check.sh`)
